# 코딩 중 겪는 혼란에 대한 이해

### 코드가 초래하는 <b>세 가지</b> 종류의 혼란

#### 1. 지식의 부족

- LTM(장기 기억 공간)의 문제 = 하드 드라이브
- 키워드의 의미

#### 2. 정보의 부족

- STM(단기 기억 공간)의 문제 = 캐시 / 메인 메모리
- 메서드나 변수의 이름과 같이 코드를 읽는 과정에서 발생하는 정보

#### 3. 처리 능력의 부족

- 작업 기억 공간의 문제 = 두뇌의 프로세서
- 코드를 읽고 처리하는 일

# 코드를 더 깊이 있게 이해하기 👍🏻

### '변수 역할' 프레임워크

(사야니에미) 11개 역할: 고정값, 스테퍼, 플래그, 워커, 최근값 보유자, 목적값 보유자, 모집자, 컨테이너, 추적자, 조직자, 임시

- 완전히 생소한 코드를 파악할 때 : 코드 프린트하거나 PDF로 저장해서 코드에 주석 달면 도움이 되는 것을 경험했다(🔎실제로 해 보기)

### 프로그램에 대한 깊이 있는 지식을 얻으려면

1. 초점을 찾는다.
2. 초점으로부터 지식을 확장한다.
3. 관련된 개체로부터 개념을 이해한다.
4. 여러 개체에 걸쳐 있는 개념을 이해한다.

### 텍스트를 읽는 것과 코드를 읽는 것은 유사하다

- 계산 능력, 즉 수학적 능력을 적용해야 하는 지식과 기술은 프로그래밍 능력에 대한 예측력이 작았다.
- 언어 능력이 더 나은 예측 요인으로 17%의 분산이 나왔다.
- 학습률, 즉 코드 아카데미의 수업을 얼마나 빨리 마쳤는지에 대한 측도는 언어 능력이 가장 큰 요인이었다. 👍🏻
- 프로그래머는 코드를 읽을 때 스캔을 먼저 한다: 30%동안 코드의 70%를 훑어보는 데 사용
- 숙련된 프로그래머: 순차적으로 읽기보다 <b>함수의 콜 스택</b> 따라 코드를 읽음

### 코드 읽기에 적용해 볼 수 있는 텍스트 이해 전력

1. 기존 지식의 활성화
2. 모니터링 : 자신이 이해한 것

- 코드를 프린트해서 이해 or not 표시하는 것도 좋은 방법
- 이해 되지 않는 부분을 표시하는 것은 자기 자신을 위해서도 유용하지만 다른 사람에게 도움 요청할 때도 역시 유용하다(🔎실제로 해 보기)
- "이 코드가 뭐를 하는지 모르겠어요" 라고 말하는 것보다 더 효과적

3. 코드에서 중요한 라인 결정하기
4. 변수명의 의미를 추론하기
5. 시각화
6. 스스로 질문하기
7. 코드 요약

# 생각의 버그

### 왜 두 번째 프로그래밍 언어가 첫 번째보다 쉬울까?

1. 학습 도중 전이(transfer during learning): LTM에 저장된 정보를 사용해서 새로운 내용을 쉽게 배움

2. 학습 전이(transfer of learning): 완전히 낯선 상황에 이미 알고 있는 내용을 적용할 때 일어남

<b> 긍정적 전이와 부정적 전이</b>

- 가브리엘 살로몬(텔아비브 대): 프로그래밍 교육의 효과가 → 다른 인지 영역으로의 전이 잘 되지 X

### 오해: 생각의 버그

#### 프로그래밍 언어를 배울 때 오개념 방지하기

1. 자신이 옳다고 확신하더라도 여전히 틀릴 수도 있다는 것을 아는 것이 중요하다. 열린 마음을 유지하는 것이 핵심이다.
2. 흔하게 발생하는 오개념에 대해 의도적으로 연구해봄으로써 그런 오개념에 빠지는 것을 방지할 수 있다.
3. 같은 프로그래밍 언어를 같은 순서로 학습한 다른 프로그래머들에게 조언을 구하는 것이다.

#### 새로운 코드베이스에서의 오개념 진단

- 오개념을 탐지하는 한 가지 방법은 두 명이 한 쌍으로 또는 더 큰 그룹으로 함께 프로그래밍하는 것이다. 다른 사람들의 생각과 가정을 알게 되면, 자신과 다르다는 것을 알게 되고 둘 중 한 사람은 잘못된 개념을 가지고 있다는 것이 분명해질 것이다.
- 특히 전문 프로그래머의 경우 자신에게 오류가 있다는 것을 깨닫기 어려울 수 있으므로 항상 코드를 실행해 보거나 테스트를 돌려봄으로써 코드에 대해 자신이 세운 가정이 맞는지 확인해야 한다.

# 코드를 작성하는 행위

### 프로그래밍 중 이루어지는 다양한 활동

#### 인지적 차원 표기법(CDN: Cognitive Dimensions of Notation)(영국, 토머스 그린)

- 디버깅은 5가지 활동을 모두 수행한다.

1. 검색 (Searching)

- 코드 내에서 검색할 때 특정 코드를 살펴본<b>이유</b>를 <b>주석문</b>으로 작성해 놓으면 도움이 된다.

2. 이해 (Comprehension)

- 코드를 읽고 실행하는 것, 테스트 코드의 실행, 리팩터링 작업

3. 전사 (Transcription)

- '단순히 코딩' 하는 활동

4. 증가 (Incrementation)

- 검색, 이해, 전사가 합쳐진 활동

5. 탐구 (Exploration)

- 코드를 사용하여 스케치하는 것과 같다.
- 코드 작성, 코드 실행, 올바른 방향으로 가고 있는지 확인하기 위한 테스트 실행, 기존 코드 읽기, 새 계획에 맞게 코드를 리팩터링하기 등

### 나는?

나는 요즘 새로 배운 기술들을 적용해 프로젝트를 진행하고 있다. 분명 강의에서 들어서 아는 내용임에도 불구하고 막상 빈 IDE에 코드를 짜 보려 하면 아무 것도 떠오르지 않아 구글신이나 필기 내용을 뒤적거리기 일쑤다. 분명 이 함수는 인강에서 들었는데, ‘왜 개별적으로는 아는데 전체적으로 조합 이 되지 않을까?’ 자책하던 중 이 책을 만나게 됐다. <br>

책 내용은 크게 4부로 이루어져 있는데, 대다수의 내용들이 이미 학부 때 교직 과정을 이수하며 들었던 내용들이라 그리 낯설지는 않았다. 큰 틀은 “교육 심리학”의 ‘정보 처리 이론’을 기반으로 전이 이론, 피아제의 인지 발달 이론 및 스키마 등을 풀어 쓰고 있고, “교육 공학”의 오수벨의 유의미 학습 이론 등도 함께 곁들이고 있는 것을 알 수 있었다.<br>

학생 때, ‘교육 심리학’ 이나 ‘교육 공학’을 배울 때는, ‘내가 교사로서 학생들을 가르칠 때 이 이론을 적용해 가르치고자 하는 내용을 얼마나 효율적으로 전달할 수 있을까?’에 초점을 맞추어 공부했었다. 책을 읽으면서 내가 이미 알던 이론들은 공부하는 학생의 입장에서, ‘내가 효과적으로 배우려면’ 이라는 조금 색다른 관점으로 바라보게 되어 매우 흥미로웠다. 동시에, 나는 학생들에게 전달할 수 있는 방법만 고민했을 뿐, 정작 "나는" 챙기지 못했다고 느껴졌다. 교육 심리학, 교육 공학 모두 좋아하던 과목들로 둘 다 A+를 받았다. 그럼에도 불구하고, 프로그래밍 공부할 때는 장기 기억에 있던 교육학 이론들이 다른 분야로 제대로된 ‘전이’가 이루어지지 않았다는 것을 스스로 깨달을 수 있었다.<br>

시간이 한정되어 있기에 정말 책에서 다루고 있는 내용들을 다 해 보면 좋겠지만, 일단 읽는 데에 더 급급해 하고 싶은 것들은 “따로 표시👍🏻”해 정리해 두었다.<br>

‘코드를 프린트하거나 PDF로 저장해서 주석 다는 거 한 번 해 보기’, ‘이해되지 않는 라인 표시’ ‘암시적 기억의 개선을 위해 이미 작성된 프로그램 수정하기’, ‘코드와 해설에서 배우기’, ‘업무 중단에 잘 대비하는 방법’ 등은 하나하나 시간날 때 꼭! 적용해 보려 한다. <br>

마지막으로, 아직 주니어로서 해야할 거는 많고, 욕심도 많은데 마치 ‘책을 찢어 먹으면 그 내용이 머릿속에 다 들어간다’ 처럼 '수주대토'와 같은 요행을 바라고 있진 않았나 반성하게 되었다. 꼭꼭 씹어서 내 것으로 만들어야 인출도 할 수 있는 것인데. 마음을 편안하게 하고, 지금부터라도 하나하나 쌓아가도록 노력해야 겠다.
